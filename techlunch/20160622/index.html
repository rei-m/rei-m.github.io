そもそもなぜ個人で作るのか

1. 新しい技術のアウトプットのため
2. 単純に自分が欲しいから
最初作ったやつ
今のやつに至るまで2回くらい大きく変わったのでその辺の話など
Ver 1.0
Language: JavaScript
Tool: Titanium Mobile
Database: SQLite
Server : None(Amazone API直叩き)
Ad : admob
最初はComicManagerという名前でリリース。
アイコンはフリー素材w(探したけどもう無かった)
リリースしてみた結果

レビューサイトで紹介されて一瞬ランキングにのる
が、すぐ失速
広告の内容も収益も微妙な感じ
Ver 2.0へ
- アプリ名を "新刊カレンダー" に。ASO大事
- アイコンをちゃんとデザイナーさんに発注(2000円くらいだったような)
- レビューで機能要望が割と来てたので対応(作者単位での検索とか先月/来月の情報とか)
- 広告はnendにした
結果
順位上がった。ブックカテゴリの無料には常時100位くらいをうろうろする感じになった
広告の収益は上がった
事件勃発

APIがタイムアウトすることが多くなってAmazonに問い合わせた結果
「そもそもお前そのアプリでAPI使うって申請してないだろうが、今すぐヤメレ」という返事をもらう
なぜこんなことがおきたのか

APIの利用申請で確かに違うサービス名を載せていたが、コメントにアプリで使うよと書いていたので大丈夫だと思っていた。申請通ったし。
一応交渉したけど断固たる態度で拒否られた。結構アフィリも売上上がってたのでwin-winなはずなんだけど。。。
悲しみの中Ver 3.0へ
- しょうがないので自分でAPI作る
- さくらVPSにサーバー構築
- Node.js/MongoDBで省エネ構成
- データはWikiからマスタ情報を。某社のサイトから発売日情報をスクレイピングで集めるように。
- スクレイピングはscriptをNode.jsで書いてcronで日次で回すようにした
結果
レビューが低くなった。順位も下がった。
当然、広告の収益も下がった
アプリの質がはAmazon先生の高性能なAPIに比例していたので、急場しのぎのAPIではやはり劣化は避けられなかった。。。
今後
- 多分閉じる。
- ATS強制でサーバに証明書つっこむのめんどくさいのとTitaniumでメンテ出来る気しないｗ
- 最近は自分が作ったやつより出来がいいのが出てきたので
